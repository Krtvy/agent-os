3.The three ways to build mobile apps00:11–00:16

Before installing anything, Nick gives you the lay of the land. There are three main approaches to building modern mobile apps. You don't have to memorise any of this — pick option 1 and ignore the rest.

Option 1 — Expo + React Native (what we'll use)

  • Uses two programming languages: JavaScript and TypeScript.
  • One codebase produces an iPhone app, an Android app, and a website at the same time.
  • Expo handles the painful "preparing the app for production" step automatically.
  • Has hot reload: when you change the code, the running app updates instantly, without you reinstalling.

Option 2 — Flutter + Firebase (Google's approach)

  • Uses Google's language called Dart.
  • Built-in design system: buttons, popups, and animations all look "very Google." Easy to assemble but limited in how custom your design can feel.
  • Tightly tied to Firebase, Google's database product. Great if you stay in Google's world.

Option 3 — Capacitor (websites in app clothing)

  • You build a regular website with HTML, CSS, JavaScript, then "wrap" it to behave like an app.
  • Easiest entry point if you already make websites — but you're optimising for everything at once, so nothing is as good as it could be.

Nick's pick: Expo + React Native. Most flexibility, most well-known, and Claude Code is good at it.

What's a framework? Pre-built scaffolding for an app. Instead of writing "what happens when the user taps a button" from scratch, the framework already knows. You just describe the unique parts.

Install Anti-Gravity

Anti-Gravity is a code editor built by Google. It looks at all your project files on the left and gives you a chat box with Claude on the right. Download from its homepage, drag to Applications, open it.

Install the "Claude Code for VS Code" extension

Any time you change the code, the app updates on your phone automatically.

The first test build

To prove everything works, Nick opens a new folder called example project, opens Claude Code there, and types: "I want to build a mobile app with Expo and React Native. Set up my workspace for me." Claude creates all the scaffolding files. Then: "Create a simple demo application, some habit tracker for demo purposes, and let me launch it on my phone."

About two minutes later: scaffolded habit tracker, QR code on screen, Nick scans, app appears on iPhone. Time from zero to running app: under 5 minutes.

What's a "local development server"? A mini-website running only on your own computer that your phone can connect to. The Expo Go app on your phone connects to this mini-website to load the app while you work on it. Think of it as a private rehearsal space before the show goes on tour.

5.The 5-part app design framework00:24–00:34

This is the most important conceptual section of the entire course. Nick argues that having Claude type code for you is the easy part — knowing what you want is the hard part. So before building anything, he forces every app through a five-step thinking exercise.

1. Core function

The one thing that, if you stripped everything else away, would still make this app what it is. For a habit tracker, it's "tap a button to record that you did the habit today." Just one sentence.

2. Core loop

An action-and-reward cycle, ideally under 30 seconds long. Every time the user does the core function, something rewarding happens — a sound, an animation, a vibration. Without this, people install your app once and forget it.

For the habit tracker: tap the habit button → hear a chime → see confetti animation → feel the phone vibrate slightly.

3. Accessory features

The supporting cast that orbits the core loop without replacing it. For a habit tracker: a history chart, the ability to create new habit types, the ability to add friends and share progress.

4. Surface area check

How many screens does the app have? If it's more than ~7, you've made it too complicated. Beginners always pile on features. Real designers cut them. Five to seven screens is the sweet spot.

5. Retention hook

Something that creates an "unfinished state" so the user has to come back. For a habit tracker: a 3-day challenge. To win, you have to open the app on three separate days. The structure itself pulls you back.

Once you've answered all five for your app, you have what Nick calls an MVP — Minimum Viable Product. The smallest version of the app worth building.

What's "MVP"? Minimum Viable Product. The smallest possible version of an app that still does its main job. You ship this first, get real users on it, then add the fancy stuff only if people actually want it.

6.Build the habit tracker (local only)00:34–01:00

Now the building begins. Nick uses a clever sequence:

Step A — Initialize a memory file with /init

Inside Claude Code, he types /init. This makes Claude examine the project and write a file called CLAUDE.md. That file is a permanent note Claude reads every future session — so Claude doesn't have to re-learn what the project is each time. Saves time and money.

Step B — Voice-dictate the full spec

Instead of typing the framework answers, Nick uses a voice-to-text tool called Aqua. He just talks: "the core function is X, the core loop is Y..." Aqua transcribes it into the chat. Way faster than typing.

Step C — Tell Claude to do it all at once

Claude initially offers to do the work in 4 phases. Nick rejects that: "No, I want you to do the whole thing, including phases 1–4. Once done, open in Chrome, not Expo, so I can test locally." He asks for everything in one go.

Five minutes later Claude has built the whole thing. Nick views it in Chrome (resized to be phone-shaped so it looks like a phone app). It works. Lots of small bugs — text is invisible in places, padding is off, the reminder feature only allows one time — but the bones are good.

Step D — Iterate via natural language

The next 20 minutes are Nick speaking detailed feedback into the voice tool: "the text colour is the same as the background colour for these icons", "the reminder only has one time", "add a developer testing menu so I can simulate completing 3 days of the challenge." He sends it. Claude fixes. He tests. Repeats.

Important point Nick makes: queue up messages. You don't have to wait for Claude to finish one task before starting the next. While Claude is fixing the design, you can already be dictating the next set of changes for the AI features. Saves a lot of waiting.

Step E — Test on real phone via Expo Go

Once Chrome looks good, Nick says "start an Expo server for this in a new terminal window". QR code appears. He scans it on his iPhone. App runs live on the phone.

This is where bugs that Chrome could never have caught show up: the haptic feedback (phone vibration) wasn't working, the icons were cut off at the top, the chime sound that was inaudible in Chrome plays properly on iPhone. Each issue gets dictated back to Claude.

What's /init? A built-in shortcut in Claude Code that creates a CLAUDE.md file describing your project. Future sessions read it automatically, so Claude doesn't have to figure out what the project is every time you open the chat.
What does "test locally" mean? Run the app only on your own computer (in a browser like Chrome made to look phone-shaped), as opposed to actually running it on a real phone. Faster and easier to iterate on. You always still need to test on a real phone before shipping.

7.Save your code to GitHub01:00–01:06

Once the app feels right, Nick has Claude push the project to GitHub. GitHub is like Dropbox for code — it stores your project online, and it also remembers every change anyone has ever made. That second part matters because when AI makes a change that breaks something, you can roll back to a working version in seconds.

He just says: "Great, create a GitHub repo for this. As well as a README. Then push." Claude does the whole thing — including adding itself as a co-author on the commit, which Nick finds hilarious.

From this point on, the project exists both on Nick's computer and on the GitHub website. He can view the code in a browser, share the link with collaborators, or restore an earlier version if anything breaks.

What's a "commit"? A saved snapshot of your code at one moment in time, with a short note saying what changed. GitHub keeps every commit forever, so you can travel back to any past version of your project.
What's a "repo"? Short for "repository" — a single project stored on GitHub. Think of it as one folder, but with the full edit history attached.

8.Add a database + login (Supabase)01:06–01:25

So far the habit tracker stores everything on the user's own phone. If you uninstall the app, your data is gone. If you switch from phone to laptop, your habits don't follow. To fix that, Nick adds two things together: a database (cloud storage for the data) and authentication (login/signup so the database knows whose data is whose).

Why both at once?

You can't have one without the other. If the database held everyone's data with no login, anyone could read anyone else's habits. So login screens and databases always ship together.

The tool: Supabase

Supabase is a free-to-start service that gives you both a database and a login system in one product. Nick says he likes it because he's used it on many apps and "you just kind of arrive at one specific stack that works pretty well." It's not the only choice — you could use Firebase or others — but Supabase is what the course uses.

What Nick actually does

  1. Goes to supabase.com, clicks "Start a project", names it "habit tracker app", picks a password, enables automatic RLS (a security setting — see jargon translator).
  2. Supabase spends a few minutes creating the database in the cloud (in Oregon, in his case).
  3. Once ready, Supabase shows him a few things: the project URL, a "publishable key", a connection string. These are all just IDs and passwords Claude needs to connect to the database.
  4. Nick copies all that info, pastes it into the Claude chat, and says "Here's all the stuff you need."
  5. Claude reads it, decides what database tables it needs (Habits, Completions, Challenges, Profiles), and creates them automatically.
  6. Nick also disables the "confirm email" setting so users don't have to click a verification link before they can log in — a friction reducer for a demo.

The result

The app now has a login screen. New users sign up, log in, and their habits sync to the cloud. Nick demonstrates by creating habits in the app, then opening Supabase's web dashboard to show those exact same habits sitting in the database — confirming the round trip works.

Local caching trick

Nick specifically asks Claude to cache data locally too. Meaning: even though the data lives in the cloud, a copy is kept on the phone so the app feels instant — no waiting for the cloud every time. The phone version syncs to the cloud in the background. Without this, every tap would feel laggy.

What's a "database"? An organised filing system that stores data and can be searched. Think of a spreadsheet that lots of apps can read and write to at the same time. Yours lives online, so any device can reach it.
What's "authentication"? The login system that lets a person prove they are who they say they are (usually by typing a password). It pairs each user with their own data so others can't see it.
What's "OAuth"? Nick says OOTH a couple of times — that's "OAuth", pronounced "oh-auth". It's the technology that lets you sign in to an app with your Google or Apple account instead of creating a new password. One click instead of a signup form.
What's "RLS" / automatic RLS? Row-Level Security. A database setting that automatically makes sure users can only see their own rows of data — they can't read other people's. Turning this on is one of the easiest, biggest wins for security. Supabase used to leave it off by default; now they ship the option to enable it on creation.

9.Add AI features (coaching & weekly reports)01:25–01:57

Now Nick adds two AI-powered features to the habit tracker:

  • Smart coaching nudges — periodic short messages like "you've nailed sleep for 14 days, but water intake is low — here's a hack."
  • Reflection summaries — longer weekly or monthly reports analysing the user's full pattern of behaviour.

The architecture (the flow of data)

Nick draws a diagram showing how this works. Don't worry about memorising it — just follow the journey:

  1. Once a day, your phone wakes up and sends a tiny request to the cloud.
  2. That request hits a Supabase Edge Function — a small piece of code that runs in the cloud only when triggered, and then turns off.
  3. The edge function reads your habit data from the Supabase database.
  4. It bundles up your data and a prompt ("write a motivating coaching message based on these habits") and sends it to Claude's API.
  5. Claude generates a custom message just for you.
  6. The message comes back to the edge function, which stores it in the database and sends it to your phone as a push notification.

Why "edge functions" instead of a server?

You could have a computer running 24/7 in a data centre waiting to do this. But that's expensive — most of the time it's just sitting idle. Edge functions are the alternative: they turn on for one second when a request hits them, do the work, then turn off. You pay per second of actual use, which for a habit tracker is pennies a month.

The API key

To let the edge function talk to Claude, you need to give it your API key — a secret password that proves you're allowed to use Claude's service and that bills usage to your Anthropic account. Nick gets one from console.anthropic.com, creates a new key called "Cal Tracker", and stores it in a special hidden file called .env so it doesn't get committed to GitHub by accident.

Iteration: making the AI output feel human

Once the system works, Nick refines the prompt. The first AI-generated nudges used em-dashes ("—") everywhere, which Nick says is a tell-tale sign of AI-written text. He tells Claude: "Modify the prompt so Claude does not output any em-dashes." Then: "Also add some emojis to the titles, specifically." A few back-and-forths and the nudges sound much more like a friend texting you and less like a corporate newsletter.

Toasts and modals

To show the messages, he asks Claude to display a "toast" — a small notification bar that slides in from the top of the screen for a few seconds, like getting a text message. (A modal is the bigger version that takes over the whole screen.)

Real push notifications

Finally, Nick tests on his real phone. The first time, push notifications need to be permitted manually in iOS Settings — Apple makes the user grant that explicitly. Once allowed, Nick triggers a notification and sees it pop up on his phone in real time. It works.

What's an "API"? Application Programming Interface. A way for two pieces of software to talk to each other. When Nick's app sends data to Claude, it does so by calling Claude's API. You can think of it like a vending machine: you put in a request, it gives you a response.
What's an "API key"? A secret password that identifies you to a service like Claude or OpenAI. Anyone who has your API key can spend your money on that service, so it must stay secret — never paste it into chats, never commit it to GitHub. Store it in a .env file.
What's a "prompt"? The instructions you give an AI. "Write a motivational message based on these habits" is a prompt. Better prompts produce better outputs. Nick iterates on the prompt repeatedly to get the tone he wants.
What's a "toast"? A small notification that slides in, stays for a few seconds, then disappears — like the banner you see when a text message arrives at the top of your phone.
What's a "modal"? A bigger popup that takes over the screen until the user dismisses it. Used for important things that need a response, like "are you sure you want to delete this?"

10.Security audit01:57–02:06

Nick says this is the step "99% of vibe coders skip" — because it's boring and not visible — but it's the difference between an app that survives and an app that gets its users' data leaked onto the dark web.

The realistic goal

You won't get to 100% secure. Even big companies don't. The goal is to fix the obvious, low-effort holes so that anyone looking for an easy target moves on to a different app instead of yours.

The technique: a single big security prompt

Nick uses a long, structured prompt (from his "Security for Vibe-Coded Apps" course) that asks Claude to scan the entire project for the most common vulnerability patterns:

  • Hard-coded secrets — API keys accidentally written directly into the code instead of stored in .env files.
  • Missing server-side validation — trusting whatever the phone sends instead of double-checking on the cloud.
  • Default-open database policies — rows readable by anyone if RLS isn't on.
  • Console error leaks — error messages that reveal too much about how the app works internally.
  • Expensive operations with no rate-limiting — features an attacker could spam to burn your API budget.

Claude outputs a tier list: Critical / High / Medium / Low for each finding. Nick says, "Run through and fix all of these errors end-to-end. After you're done, test and ensure they're 100% solved." Claude fixes them.

The two-pass trick

Then — and this is the clever bit — Nick clears the conversation and runs the entire security prompt again, with no memory of the first run. Why? Because fixing one problem sometimes creates a new one. A fresh second pass catches anything the first one introduced. Most of the original "Fails" are now "Passes" or "Partials." Nick asks Claude to also fix the Partials.

By the end, the app isn't bulletproof but is solid enough that "no attacker is going to bother — they'll go find an easier target."

What's "CWE"? Common Weakness Enumeration. A standardised numbering system for known security weaknesses ("CWE-89: SQL injection", "CWE-79: Cross-site scripting", etc.). Each finding in the audit gets tagged with its CWE number so you can look up exactly what the weakness is.

11.The framework as a repeatable checklist02:06–02:13

Before moving on to the next app, Nick zooms out and draws the whole process as a flowchart. This is the checklist he'll repeat for every app he ever builds, in order:

  1. MVP ideation — answer the 5 framework questions.
  2. Build — dictate the MVP to Claude, let it scaffold.
  3. Design — once it works, polish the look. Tell Claude things like "make this feel more like an Apple app", "use a serif font", "darker blue colour scheme."
  4. Test 1: Chrome — view the app in a phone-sized Chrome window, find bugs.
  5. Test 2: phone mirror — use iPhone Mirroring on a Mac to test on the phone while seeing the screen on your computer.
  6. Test 3: real phone — pick up the phone and use it as a phone. Catches things that only matter physically: haptic feedback, push notifications, thumb reach.
  7. Add database & login — only after the local version works.
  8. Test 1 + 2 + 3 again — because adding the database introduces new failure modes.
  9. Security audit — two passes.
  10. Final test — one more end-to-end check.
  11. Deploy — submit to App Store.

He emphasises: this loop is the same for every app. The reason he's about to build two more apps isn't to teach you more — it's to drill the pattern.

12.Build app #2 — calorie tracker02:13–02:55

The second app is a calorie tracker similar to Cal AI (which sold for $50–100M to MyFitnessPal). You take a photo of your food, AI estimates the calories and macronutrients (protein, carbs, fat), you log it.

What's new in this build

1. Photo input via Anthropic's vision API

The user's photo gets sent to Claude, which is told to analyse the food in the image and return a JSON object with estimated calories, protein, carbs, and fat. Plus or minus 10–15% accuracy, but good enough for daily tracking.

2. Personalised goals via the Mifflin-St Jeor equation

The app asks the user their age, height, weight, sex, and activity level, then uses a standard formula (the Mifflin-St Jeor equation, a textbook calculation for resting metabolic rate) to recommend a daily calorie target.

3. Screenshot-driven design

This is the most useful trick in the whole video. Instead of describing the design Nick wants in words, he:

  1. Goes to Dribbble (a website where designers post their work) and finds time-tracker app screenshots he likes.
  2. Screenshots them.
  3. Drags the screenshots into Claude Code and says "I love the design of this app. I want you to start by emulating it."

Claude can see images. It pulls colour palettes, spacing, font choices, layout — and adapts them to the new app.

4. A custom colour palette via the UI Colors tool

For finer control, he uses uicolors.app to pick a specific blue-grey palette, then copies the exact colour codes into Claude.

5. Anthropic API key for the photo analysis

Same setup as before. New key created on the Anthropic console, pasted into .env.

Iteration patterns Nick shows

  • Test loops queue up — while waiting for one fix, dictate the next round.
  • "Run through every page yourself" — Nick asks Claude to open each page in Chrome's developer tools, take a screenshot, list every alignment / spacing / colour inconsistency, and then fix them in a loop. This works because Claude can now look at its own work.
  • Compaction — once the conversation gets very long (around 250,000 tokens), Nick types /compact. Claude rewrites the whole conversation history into a much shorter summary, freeing up space for more work without losing context.

Sign-in page lifted from Dribbble too

For the login screen, Nick searches Dribbble for sign-in pages he likes, screenshots one, and tells Claude to build something like it. Result: the calorie tracker has tiny food-and-weight icons floating around the sign-in page, very polished, in minutes.

What's "Dribbble"? A community website where designers post screenshots of their app designs. It's basically Pinterest for app design. Great for inspiration. dribbble.com
What does "/compact" do? A Claude Code shortcut that rewrites your entire current conversation into a much shorter summary, keeping the meaning but reducing the size. Useful when conversations get long and slow — Claude works faster on smaller contexts.
What's a "token"? Roughly speaking, a small chunk of text — a word, a piece of a word, or a bit of code. AI services bill by tokens. The longer the conversation, the more tokens it uses, and at some point performance degrades. Compaction shrinks the count.

14.Build app #3 — the Pomodoro forest03:05–03:43

The third app is a Pomodoro timer (a productivity technique where you work in 25-minute focused blocks). Nick's twist: every time you complete a Pomodoro, a tree grows. Over time you build a forest. Skip your Pomodoros and your trees start withering.

This app is the most visual of the three and shows off design techniques you haven't seen yet.

The new technique: AI-generated art

Nick uses GPT Image-2 (OpenAI's image generation model) to generate concept mockups for his app. He asks Claude to describe his app in words → he feeds that description to GPT Image → GPT Image returns four mockup screens with hand-drawn-style trees, fields, and skies.

Then he asks Claude to extract the individual trees, backgrounds, and sky elements from the mockups as separate image files. He saves them in an images/ folder in the project. Claude uses these real images directly in the app instead of trying to draw them in code.

Other new bits in this build

  • Plan mode — for this build, Claude switches itself into "plan mode" and asks Nick clarifying questions before writing code: "should the tree grow in real time as the timer ticks down, or play a pre-made animation at the end?" Nick picks each option.
  • Sub-agents — Claude spawns a small helper that does background research on "React Native animation libraries" while the main thread keeps working. The helper reports back, then disappears.
  • Haptic feedback at the design level — Nick asks for it explicitly: "when I tap on a seed, it should jiggle slightly and the phone should buzz."
  • Withering animations — neglected trees visually wilt; the app sends push notifications inviting you back.

Why so much polish on such a simple app?

Nick's point: the Pomodoro core function (a timer) is trivial — your phone already has a timer. The reason a user would choose this app is the feeling. So the entire app is engineered around the "growing tree" payoff. Without that, you have nothing.

What's "plan mode"? A mode Claude Code can enter when a task is complex enough that it wants to ask clarifying questions before charging ahead. You'll see "PLAN" on the screen and Claude will present numbered options. You either pick from the suggestions or override.
What are "sub-agents"? Smaller, focused versions of Claude that the main Claude can dispatch to handle a specific question — like researching a library, or testing one page of an app — without cluttering the main conversation. They report back and disappear.
What's an "SVG"? Scalable Vector Graphic. An image format described as code, not pixels. The advantage: it scales to any size without getting blurry. Used a lot for icons.

15.Submit to the App Store03:43–04:00

Final stretch. Nick picks the Calorie Tracker (most complex of the three) and walks it all the way through to App Store submission.

What you need before you can submit

  1. An Apple Developer account — sign up at developer.apple.com/account. Costs $99/year. Without this you cannot publish to the iOS App Store.
  2. A Google Play Developer account — sign up at play.google.com/console/signup. One-time $25 fee. For the Android side. (Optional if you're iOS-only.)
  3. A working app — fully tested on Chrome, mirror, and real phone, plus security audit passed.
  4. A privacy policy hosted on a real website. Apple requires this. Nick has Claude write his — fed the Apple submission rules to Claude and got a complete privacy policy hosted at leftclick.ai/caltracker/privacy in minutes.
  5. A support page hosted on a website. Same trick.
  6. App screenshots in specific sizes for iPhone and iPad. Claude can take these by opening the app in Chrome and screenshotting at the right dimensions.
  7. App store text — promotional copy, full description, keywords. All written by Claude.

The build-and-submit flow

  1. Prepare for production — Claude generates two config files (app.json and eas.json) that describe the app to Apple: name, version, icon, permissions needed.
  2. Build via EAS — Nick runs the command Claude gives him. EAS (Expo Application Services) is Expo's cloud building service. You upload your code; their servers compile it into a finished iOS app package. Free tier takes longer; paid tier is faster.
  3. Submit via EAS — once built, run eas submit --platform ios. It uploads the finished build directly to Apple's App Store Connect.
  4. App Store Connect — log in at appstoreconnect.apple.com. Drag in screenshots, paste promotional text, set price (free or paid), select category, fill in age ratings, content rights, in-app-purchase declarations, and so on. This is tedious but Claude can produce all the text and images.
  5. Test sign-in for reviewers — Apple requires you to give them a username and password for an actual account in your app so a human reviewer can log in and test the features. Nick uses something like test@gmail.com / t3stx!.414.
  6. Submit for review — top-right corner of App Store Connect. Apple reviews within a few days. They might reject and tell you what to fix, or approve.

TestFlight (optional but recommended)

Apple has a service called TestFlight that lets you share your app with up to 10,000 testers privately, before submitting to the public App Store. Nick recommends spending 1–2 weeks on TestFlight with real users to find bugs before going public. He doesn't do this on camera because of video length, but flags it as best practice.

What's "EAS"? Expo Application Services. The cloud-build part of Expo. It takes your JavaScript code and turns it into the binary file (.ipa for iPhone, .aab for Android) that Apple and Google actually require. You'd otherwise need a Mac with Xcode installed for iOS builds — EAS removes that requirement.
What's "TestFlight"? Apple's beta-testing service. Lets you share an in-progress app with specific people via a private link, without going through the full public App Store review. Great for catching bugs from real users before going public.
What's "App Store Connect"? Apple's web dashboard where you actually manage your app's listing: screenshots, descriptions, pricing, reviewer notes, submission status. Distinct from the developer account itself — it's where the per-app paperwork lives.

Jargon Translator — every technical term, defined plainly

Sorted alphabetically. If a term appears in the video and isn't in this list, it was a passing mention that doesn't really matter.

Android
Google's phone operating system. Used by Samsung, Google Pixel, OnePlus, and most non-Apple phones.
Anti-Gravity
A code editor (the app where you write and edit your project's files) built by Google. Nick uses it because it's similar to VS Code but supports Claude Code natively.
API
Application Programming Interface. A way for two pieces of software to communicate. When your app sends a food photo to Claude for analysis, it's "calling Claude's API."
API key
A secret password that identifies you to an API (like Claude's). Keep it secret — anyone with it can spend your money. Store in .env files, never paste into chats, never commit to GitHub.
App Store Connect
Apple's web dashboard for managing your app's listing on the App Store. Screenshots, description, price, reviewer login, submission status all live here.
Apple Developer account
The $99/year membership you need to publish apps to the iOS App Store. Sign up at developer.apple.com/account.
Authentication
The login system. Lets users prove who they are, usually with a password or a Google/Apple sign-in. Pairs each user with their data.
Cache (local caching)
Keeping a copy of data on the user's own device for speed. The app reads from the cache instantly while syncing to the cloud in the background. Without caching, every tap waits on the cloud.
Capacitor
A framework that lets you take a regular website and wrap it so it behaves like a mobile app. Easier to learn but less flexible than React Native.
CD (change directory)
A terminal command that moves you into a folder. cd ~/projects/habit-tracker takes the terminal to that folder.
Claude / Claude Code
The AI assistant from Anthropic. Plain "Claude" is the chat. "Claude Code" is the version that can actually read and write files on your computer.
CLAUDE.md
A plain-text memory file at the root of your project. Claude reads it at the start of every session, so it remembers what your project is. Created automatically by typing /init.
Commit
A saved snapshot of your code at one moment, with a short message describing what changed. GitHub remembers every commit forever.
Compaction (/compact)
A Claude Code shortcut that compresses the current conversation into a shorter summary. Use when conversations get very long and Claude starts to slow down.
CWE
Common Weakness Enumeration. A standardised numbering system for known security weaknesses. The security audit tags each finding with its CWE number.
Dart
Google's programming language, used with the Flutter framework. We don't use it in this course.
Database
Organised online storage. Like a giant spreadsheet that lots of apps can read and write to at the same time. Lives in the cloud so any device can reach it.
Deployment
Pushing your app to a place where real users can install it (the App Store, the Play Store, a public website). The final stage of the build pipeline.
Dribbble
A website where designers share screenshots of their work. Great place to find design inspiration. dribbble.com
EAS (Expo Application Services)
Expo's cloud-build service. Takes your code and produces the finished iOS / Android app file that Apple and Google require. Removes the need for a Mac with Xcode.
Edge function
A small piece of code that runs in the cloud only when triggered, then turns off. Cheaper than an always-on server. Supabase Edge Functions are the version used here.
.env file
A hidden file at the root of your project where you store secrets like API keys. It's automatically excluded from GitHub uploads via the .gitignore file.
Expo / Expo Go
Expo is the toolkit Nick uses to build apps. Expo Go is the free phone app from the App Store / Play Store that lets you run your in-progress app on your real phone by scanning a QR code.
Firebase
Google's database and login product. Direct competitor to Supabase. Nick uses Supabase instead because of personal preference.
Flutter
Google's app-building framework. Uses Dart. Has its own opinionated design system that makes Android apps look "very Google." Nick chose React Native over Flutter for this course.
GitHub
Online storage for code, plus a complete history of every change. Like Dropbox-for-code with a time machine attached.
Haptic feedback
The little vibrations your phone produces when you tap certain buttons. Real, deliberate UX feature — not a quirk. Adding haptics to your app makes it feel professional.
Hot reload
When you change the code, the running app on your phone updates instantly without reinstalling. Saves enormous time during development.
HTML / CSS / JavaScript / TypeScript
The four core languages of the web. HTML is the structure, CSS is the styling, JavaScript is the behaviour, TypeScript is JavaScript with type-checking added. Expo + React Native use JavaScript and TypeScript.
/init
A Claude Code shortcut that creates a CLAUDE.md file describing your project so future sessions don't start from scratch.
iOS
Apple's phone operating system. The thing that runs on iPhones.
JSON
A simple text format for structured data. Looks like {"calories": 650, "protein": 42}. The standard way apps and APIs trade data.
JWT
JSON Web Token. A specific kind of login token that proves a user is signed in. Mentioned in passing during the AI features section.
Local-first
An approach where the app's local cache is the primary source of data and the cloud is the backup, instead of the other way round. Results in apps that feel instant.
Local storage
Data stored on the user's own device only. Doesn't sync between phone and laptop. The version of the habit tracker before Supabase was added used local storage only.
Mifflin-St Jeor equation
A standard physiology formula for estimating how many calories someone burns at rest, based on their age, height, weight, and sex. Used by most calorie-tracker apps.
Modal
A popup that takes over the whole screen until the user dismisses it. Used for important interruptions: confirmations, errors, sign-in screens.
MVP (Minimum Viable Product)
The smallest version of an app that still does its main job. Ship this, then iterate based on real users.
OAuth (pronounced "oh-auth")
The technology that lets users sign in with their Google or Apple account instead of creating a new password. Nick keeps saying "OOTH" — he means OAuth.
Plan mode
A Claude Code mode where Claude asks clarifying questions before writing code. Triggered automatically when a task is complex enough to need decisions.
Pomodoro
A productivity technique: work in 25-minute focused blocks ("Pomodoros") separated by 5-minute breaks. Builds focus over time.
Postgres / PostgreSQL
The specific database technology Supabase uses under the hood. You don't need to know anything about it directly — Claude handles it.
Production
The "real" version of your app that real users will install — as opposed to the in-development version you've been playing with. "Preparing for production" means locking down the configuration before publishing.
Prompt
The instructions you give to an AI. Better prompts produce better outputs. Nick rewrites prompts repeatedly to tune the AI's behaviour.
Push notification
A message that appears on a user's phone even when your app is closed — like getting a text. Requires the user to grant permission in iOS / Android settings first.
QR code
The black-and-white square pattern. Your computer prints one; you scan it with your phone's camera to open the in-development app via Expo Go.
React Native
Facebook / Meta's framework for building phone apps with JavaScript. The framework underlying everything in this course.
Repo / repository
A single project on GitHub. One folder of files plus all its edit history.
RLS (Row-Level Security)
A database setting that automatically restricts each user to only their own rows of data. One of the easiest, biggest wins for security. Enable it when creating a Supabase project.
Schema
The "shape" of your database — what tables exist, what columns are in each table, what types of data they hold. Claude designs and applies the schema for you.
Serif vs sans-serif fonts
Serifs are the little tails at the ends of letters in fonts like Times. "Sans serif" means "without serifs" — fonts like Helvetica. Nick experiments with both.
Server
A computer in a data centre that responds to requests over the internet. The Supabase database lives on Supabase's servers.
Serverless
The opposite of "always-on server." Code that only runs when triggered, then turns off. Supabase Edge Functions are serverless. You pay per second of actual use, not 24/7.
SQL
The language used to talk to databases. Nick mentions it but you almost never have to write it yourself — Claude does.
Splash screen
The brief logo screen you see when you first launch any app. Needs to be created for the App Store submission. Claude can make one.
SVG (Scalable Vector Graphic)
An image format described as code, not pixels. Scales to any size without blur. Used for icons and simple illustrations.
Supabase
The database + login service used in this course. Free to start. Cheap and reliable. Lives at supabase.com.
Sub-agent
A smaller, focused version of Claude that the main Claude can dispatch to handle a specific question (research, testing) and then disappear. Keeps the main conversation clean.
Terminal
The black-and-white text window where you type commands directly to the computer. On Mac it's called "Terminal." Claude can drive it for you.
TestFlight
Apple's private beta-testing service. Share your in-progress app with up to 10,000 people before submitting to the public App Store.
Toast
A small notification that slides in, stays for a few seconds, then disappears. The banner you see when a text arrives.
Token
The unit of measurement for AI usage. Roughly a word or piece of a word. Long conversations use more tokens. AI services bill by tokens.
UI / UX
UI is User Interface — what the app looks like. UX is User Experience — how it feels to use. Different but related concerns.
VS Code / Visual Studio Code
Microsoft's free code editor, the most popular one in the world. Anti-Gravity is built on top of it.
WebFetch
Not mentioned in the video, but it's the Claude Code tool that lets Claude read web pages — which it can do automatically when it needs to research a library.
YAML / Markdown / .md
Plain-text formats for configuration and documentation. CLAUDE.md is a Markdown file. Don't worry about the distinction.
